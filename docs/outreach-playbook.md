# Outreach Playbook

Default proof URL:

- `/case-study/pain-tracker`

Use this page as the primary link for outbound messages. Do not lead with the homepage unless someone asks for the broader service surface.

## Core message

Use this for direct email or DM to a technically literate founder or technical lead.

```text
Subject: trust hardening for [product/company]

You are post-MVP and handling sensitive user data, so the main question is no longer whether the product works. It is whether it stays usable and defensible when trust, connectivity, or operator confidence start to fail.

I do trust hardening work for sensitive-data products: reducing dangerous data collection, hardening critical flows, and producing a decision-grade roadmap teams can actually ship.

This case study is the clearest example of the work:
[case-study-url]

If the fit is real, send repo + context + constraints and I can tell you quickly whether a Trust Hardening Review makes sense.
```

## Short DM version

Use this in founder DMs, GitHub messages, or short follow-ups after a public interaction.

```text
You are building in a category where trust failure gets expensive fast. I do trust hardening for post-MVP sensitive-data products.

Best proof link:
[case-study-url]

If useful, send repo + context + constraints and I can tell you quickly whether there is a fit.
```

## Comment-thread follow-up

Use this after someone has already engaged with a post or technical comment.

```text
I wrote up one concrete example of the kind of hardening work I mean here:
[case-study-url]

It shows the starting condition, trust risks, architectural decisions, hardening moves, and resulting posture.
```

## Warm intro reply

Use this when someone is already aware of the work and only needs the most direct next step.

```text
Best single link is this case study:
[case-study-url]

If the operating constraints look familiar, send repo + context + constraints and I can tell you whether a Trust Hardening Review is the right first move.
```

## How to use it

- Replace `[case-study-url]` with the production URL for `/case-study/pain-tracker`.
- Replace `[product/company]` only when you have a real reason to personalize.
- Keep the first message short. The page should do the heavy lifting.
- Default to one link only: the case study.
- Only send the homepage or proof page if the recipient asks for the broader surface.

## What to watch

- Replies that ask about scope, timing, or fit are positive signals.
- Silence after a click usually means the case study did not create enough urgency or fit clarity.
- Clicks with no contact action suggest the page may need a sharper next move.
- Repeated questions about pricing or engagement shape suggest the front-door offer still needs tighter qualification.
